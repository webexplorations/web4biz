CSC235 - Project: Process

Project Summary

Practicing Product Development

As you worked through the learning activities for this week, you learned about several best practices used during application development.

Now it is your turn to put the process to work.

Imagine attending a meeting where a client just hired you to create a major software project that you estimate to be worth over $6,000. Where will you start?

This project should be completed as five distinct steps:

  1. Do a needs assessment.
  2. Do a problem analysis.
  3. Write proof-of-concept code.
  4. Prototype the UI.
  5. Write the app.

Specifications (50 points total possible)

Step 0. Think of a project

Think of a simple app that will solve a problem similar to the hyperlink problem presented in this module. Here are some examples to act as a catalyst for your own app:

  1. A restaurant menu with links for different meals and/or courses, but with details that need to be updated on a regular basis. A visitor to the website can click on a link such as "Daily Special" or "List Soups/Salad/Entrees" to open a list of the current offerings for that category. A school lunch menu could be a variation of this type of app.
  2. A university course schedule that needs to be updated regularly to display available courses, as well as details for time, place, and instructor.
  3. A work schedule that allows employees to click their names revealing the times they are scheduled to work that week.

Step 1. Needs Assessment (10 points)

Create a web page named presentation.php containing at least five questions you would ask a client in order to complete this project. Then, acting as the client, answer these questions.

As a guide, you can start with the five Ws (Who, What, When, Where, and Why). Followed up by any other information that would be helpful to fully understand this project.

  • Design this page as a presentation to a potential high-end client. (Your instructor is your high-end client in this case.)
  • Keep UX and readability in mind using headings, bullet points, bold and regular, and other HTML/CSS techniques. For example, display the questions using bold text and the answers as regular text.
  • Include your name, the title you are giving your project, and the current date at the top of the page. This information should display. Do not hide this information as a comment in the code.
  • Include a standard coding comment block at the top of the file
  • Include links near the bottom of the page to the page to the following on your remote server:
    • presentation.php file.
    • The home page (index.php).
    • The editing or management page (edit.php).

Step 2. Problem Analysis (10 points)

At the bottom of presentation.php describe at least two ways you could approach this application as the developer. Your suggestions can be variations on the same approach. Your solution should break down the problem into individual parts.

Decide which is the best approach for your client and write a short paragraph highlighting your recommendation and why you think this is the best option for your client.

You may find it helpful to answer the following questions:

  • How will the data be handled? What are some alternative formats for the data?
  • What will the UI be like, from when the user clicks on a link to when information displays? Visualize this using a simple sketch or flowchart. (A sketch is worth a thousand and one words to every client!)
  • Make this information appealing and readable as if you will presenting these ideas to your client.

Step 3. Proof-Of-Concept Code (10 points)

Using your problem analysis, documented in the presentation.php file, write proof-of-concept code demonstrating the solution(s). Name this file edit.php. The output from this code does not have to be pretty!

This code is to help you noodle around, solving specific problems inherent in the project.

  • Determine where the sticky-wicket(s) (specific problem areas) are in the project specs. What techniques will you need to figure out to make sure these parts of the program work?
  • Keep the code as simple as possible. If you can solve the problem in a single line of code, that's excellent.
  • Write several small, test programs to test specific solutions or techniques. Then, once you have a piece working, add it to the edit.php program as a function. This will allow you to focus on one thing at a time, building the final solution, one piece at a time. (Psychologically, this is very hard for new programmers to do. But, once you get in the practice of writing "side" programs, you will never go back to the old way of only working one large and forever growing page of code.)
  • Do not spend time writing user input code. Simply set up a variable with test data and use it in your example code. Later, it will be easy to replace that variable with data actually collected from the user.
  • Focus on each problem without worrying about the other challenges. Being able to zoom in and focus on a single programming issue is key to being an effective programmer.
  • Name your test files beginning with the word "test" in the name. To reduce clutter put all these files in a folder named testCode

Step 4. Prototype the UI (5 points)

Draw a prototype screen (wireframe) of your app based on the client's specs. You do not have to write the actual code. Just show a visual picture of what the screen might look like.

  • Use any drawing software you with to create this wireframe. Or, use pencil and paper and scan in the image.
  • Include this image at the bottom of prjReadMe.php Verify that you have formatted the image appropriately for display on the web page for size and clarity. Tiny, unreadable images or fuzzy images are not acceptable. As a rule of thumb, images should be between 500px and 1000px wide, with a proportionate height.
  • You will need a wireframe for both the user screen as well as the editing screen, used to update the links.

Step 5. Write the App (10 points)

Write the final app. It should include the following:

  • Name your app index.php with links to variable content, similar to what was done in the lessons. Each piece of variable content should have a unique ID. The content does not have to be a hyperlink.
  • A separate web page named edit.php with a user-friendly interface that would allow an office manager or administrative assistant to update the variable content.
  • edit.php should also display a unique ID for each piece of content and the corresponding value for that content (similar to the ID/URL combination demonstrated in the lessons.)
  • All pages must have a clean UI, using the same CSS code for all pages.

Other Notes

Points will be deducted for the following:

  • Not following the naming conventions (code, file and folder names, form elements, data field names)
  • Using filenames other than those specified.
  • Spelling or grammar errors. Proof-read each others work. Use Grammarly.com
  • Inadequate commenting.
  • Missing a comment block at the top of each page of code.
  • Poorly formatted code. Use tools such as Free Formatter or Dan's Tools - HTML formatter

Commenting

Comments are required for each block of code. Add comments as notations to the instructor showing your knowledge and understanding of the code in your own words. Just having the code is not acceptable.

Every file created by you must have a comment header block at the top of the page including the following:

  • Name of the file and its purpose.
  • Course name and project name
  • Your name and email.
  • The date the file was written/created.
  • A place to document future revisions. (date, person, what was done)

Naming Conventions

The following naming conventions must be followed when naming all folder, filename, id, and class names:

  • Always start with a lower case letter.
  • Do not use spaces or underscores "_"
  • useCamelCaseForReadability. Acronyms can remain all caps, using a hyphen "-" only when absolutely necessary.
  • Use singular names. For example, graphic not graphics.
  • Keep all HTML elements and filename extensions lower case (Watch out for photo editing software that tries to save the filename with the extension set to all caps.)
